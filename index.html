pprof

Posted on 2022-09-07 Edited on 2022-09-08

1. 初识

profiling，程序执行过程中，收集能够反映程序运行状态的数据，是一种动态研究程序行为的方法

go自带的pprof库可以用来分析程序的运行情况，包括两种：

  • runtime/pprof

运行时间一定

  • net/http/pprof

一直运行以提供服务

实际上也是调用runtime/pprof，封装成接口提供网络访问

pprof可提供性能数据：

  • allocs

内存分配

  • blocks

阻塞操作

  • cmdline

显示程序启动命令及参数

  • goroutine

当前所有协程的堆栈信息

  • heap

堆上内存

  • mutex

锁争用情况

  • profile

cpu占用情况

  • threadcreate

系统线程创建情况

  • trace

程序运行追踪信息

备注：

allocs和heap采样信息一致，不过前者是所有对象的内存分配，后者是活跃对象的内存分配 （何为活跃对象？）

2. 使用

2.1 runtime/pprof

类似的还有启动内存profile等等

2.2 net/http/pprof

pprof会自动注册handler，处理相关请求

第一个路径 /debug/pprof/ 下面其实还有 5 个子路径：

goroutine
threadcreate
heap
block
mutex

启动服务后，可以直接在浏览器中访问：

http://ip:port/debug/pprof

得到汇总页面

可以直接点击上面的链接，进入子页面，查看相关的汇总信息。

关于 goroutine 的信息有两个链接，goroutine 和 full goroutine stack dump，前者是一个汇总的消息，可以查看 goroutines 的总体情况，后者则可以看到每一个 goroutine 的状态。页面具体内容的解读可以参考【大彬】的文章。

点击 profile 和 trace 则会在后台进行一段时间的数据采样，采样完成后，返回给浏览器一个 profile 文件，之后在本地通过 go tool pprof 工具进行分析。

KAFKA

Posted on 2022-08-15 Edited on 2022-08-17

1.简介

三个关键功能：

  1. 发布（写入）和订阅（读取）事件流，包括从其他系统持续导入/导出数据 。
  2. 根据需要持久可靠地 存储事件流。
  3. 在事件发生时或回顾性 地处理事件流。

疑问：

生产者一旦写入就会持久存储，消费后也不会删除，直到设置的过期时间

同一份数据是否可被多个消费者同时消费，那么各消费者之间的关系又是什么？互不影响，甚至感知不到其他消费者的存在？

对某个topic来说，多生产者 多消费者

kafka的性能是恒定的，与数据大小无关

能够保证给定topic分区的任务消费者都能以与事件写入完全相同的顺序读取该分区的事件

2. 部署架构及概念

标准化与现有其他数据系统的集成，简化连接器的开发、部署和管理

kafka stream API

broker：kafka实例

partition：

一个topic可以有多个分区，一个分区可以由多个副分区。

每个记录由键值以及时间戳组成

默认情况下，通过轮询将消息发送到不同的分区上；在某些情况下，生产者会把消息直接写到特定的分区，通常通过消息键和分区器来实现，分区器为键生成一个散列值，并将其映射到指定分区，可以保证同样的键映射到同一个分区上（有什么应用场景？）

生产者也可以自定义分区器，根据不同的业务规则将消息映射到分区（为什么要这么做？应用场景？）

同一个topic可以被多个消费者消费，消费者把每个分区最后读取的消息偏移量保存到zookeeper或者kafka上，那么是否支持多个消费者消费同一个分区？如果支持，那么偏移量是消费者级别的？也就是说针对每个消费者都会记录一个偏移量？不然不同消费者不同的消费速度如何区分？

还有一点，不同分区的数据应该是不一样的，那么消费者必须消费所有分区才能处理所有数据，那么消费者组概念随着而出？

partition的数量与topic绑定？不同topic的partition任意设定？

无法整个topic范围内保证消息的顺序，只能保证单个分区下的消息顺序，需要严格保证消息顺序的话，需要将partition设为1

通过partition实现数据冗余和伸缩，数据冗余表现在同一个partition可以有多个复制品

每个partition会有一份或多份数据保存，当有多份数据时，会有一个首领副本，首领副本负责消息的读写处理，其他副本只是从首领副本复制消息，不参与请求处理，若首领副本down掉，会重新选举首领副本，那么原首领副本上线后作为其他副本是否会将现首领副本的消息同步过来

TLS

Posted on 2022-07-04 Edited on 2022-07-05

1. 理论

1.1 密钥交换算法：

1.1 RSA

  1. 第一次 （client->server）

1.1 Client Hello

消息内容：client使用的TLS版本号、支持的密码套件列表、随机数1

  1. 第二次 （server->client）

2.1 SayHello

消息内容：确认TLS版本号，从client的握手消息中的加密套件列表中选择一个，随机数2

Note：加密套件解释：“Cipher Suite: TLS_RSA_WITH_AES_128_GCM_SHA256”

即：握手时密钥交换算法和签名算法都是使用 RSA; 握手后的通信使用 AES 对称算法，密钥长度 128 位，分组模式是 GCM；摘要算法 SHA256 用于消息认证和产生随机数；

2.2 Server Certificate

server的证书

2.3 Server Hello Done

内容为空，告诉client server端打招呼结束

  1. 第三次 （client->server）

3.1 Client Key Exchange

验证客户端证书（若tls config配置skipVerify则不必验证；若tls serverName不为空，则对比serverName与server证书中的commonName或者SAN是否一致）

Note：go 1.15开始弃用commonName，要求SAN

随机数3，又名pre-master，并用server的公钥加密该随机数传递给server

3.2 ChangeCipher Spec

告诉server开始使用加密方式发送消息，即以上都是明文传送

3.3 EncryptedHandshake Message（Finished）

之前的数据计算摘要，再用计算出的密钥（master secret）加密发送到server， 让服务器做个验证，验证加密通信是否可用，以及验证之前握手信息是否有被中途篡改过

密钥（master secret）由 randomClient、randomServer、 pre-master三个随机数计算得出

  1. 第四次（server->client）

4.1 ChangeCipher Spec

类似client

4.2 Encrypted Handshake Message

类似client
